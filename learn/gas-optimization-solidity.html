Gas Optimization in Solidity

Estimated reading time: 12 minutes

Gas costs are one of the most critical considerations when developing smart contracts on Ethereum and EVM-compatible blockchains. Every operation in a smart contract consumes gas, and inefficient code can result in prohibitively expensive transactions for users. This article explores practical techniques for optimizing gas consumption in Solidity smart contracts, helping developers write more efficient and cost-effective code.

Understanding Gas in Ethereum

Gas is the unit of computational effort required to execute operations on the Ethereum Virtual Machine (EVM). Each opcode has a fixed gas cost, and the total gas consumed by a transaction determines its cost. Key concepts include:

  • Gas Price: The amount of ETH a user is willing to pay per unit of gas, measured in gwei.
  • Gas Limit: The maximum amount of gas a user is willing to spend on a transaction.
  • Block Gas Limit: The maximum total gas allowed in a single block, which constrains transaction complexity.

Understanding the EVM's gas cost structure is essential for optimization. Storage operations (SSTORE, SLOAD) are among the most expensive, while memory and stack operations are relatively cheap.

Storage Optimization

Storage is the most expensive resource in the EVM. Writing to storage costs 20,000 gas for a new slot and 5,000 gas for updating an existing one. Reading from storage costs 2,100 gas (cold) or 100 gas (warm, after first access in a transaction).

Variable Packing

The EVM uses 32-byte storage slots. Multiple smaller variables can be packed into a single slot to reduce storage costs:

// Unoptimized: uses 3 storage slots
uint256 a;
uint256 b;
uint256 c;

// Optimized: packs into 1 storage slot
uint128 a;
uint64 b;
uint64 c;

Solidity automatically packs variables declared consecutively if they fit within a single 32-byte slot. Order your struct fields and state variables by size to maximize packing.

Minimize Storage Writes

Cache storage variables in memory when reading them multiple times within a function:

// Unoptimized: reads from storage 3 times
function calculate() public view returns (uint256) {
    return storedValue * storedValue + storedValue;
}

// Optimized: reads from storage once
function calculate() public view returns (uint256) {
    uint256 cached = storedValue;
    return cached * cached + cached;
}

Use Mappings Over Arrays

Mappings are generally more gas-efficient than arrays for lookups. Arrays require iteration and have additional length storage overhead, while mappings provide O(1) access.

Function Optimization

Use External Over Public

For functions that are only called externally, use the external visibility modifier instead of public. External functions can read arguments directly from calldata, which is cheaper than copying to memory as public functions do.

Short-Circuit Evaluation

Place the most likely condition to fail first in require statements and conditional expressions. Solidity uses short-circuit evaluation, so if the first condition fails, the remaining conditions are not evaluated:

// Put cheap and likely-to-fail checks first
require(msg.sender == owner, "Not owner");
require(balances[msg.sender] >= amount, "Insufficient balance");

Use Custom Errors

Custom errors (introduced in Solidity 0.8.4) are more gas-efficient than revert strings because they use less bytecode and don't store string data:

// Expensive: stores string in bytecode
require(balance >= amount, "Insufficient balance");

// Cheaper: custom error
error InsufficientBalance(uint256 available, uint256 required);
if (balance < amount) revert InsufficientBalance(balance, amount);

Loop Optimization

Unchecked Arithmetic

In Solidity 0.8+, arithmetic operations include overflow/underflow checks by default. For loop counters that cannot realistically overflow, wrapping the increment in an unchecked block saves gas:

for (uint256 i = 0; i < length;) {
    // loop body
    unchecked { ++i; }
}

Cache Array Length

Reading array length from storage in every iteration is wasteful. Cache it in a local variable:

uint256 len = myArray.length;
for (uint256 i = 0; i < len;) {
    // process myArray[i]
    unchecked { ++i; }
}

Data Types and Operations

Use bytes32 Over string

For fixed-length data (like identifiers or hashes), bytes32 is significantly cheaper than string because it fits in a single storage slot and doesn't require dynamic memory allocation.

Use Prefix Increment

++i is slightly cheaper than i++ because it doesn't create a temporary variable to hold the original value.

Bitwise Operations

Bitwise operations are cheaper than arithmetic equivalents. For example, multiplying or dividing by powers of 2 can use bit shifts:

// Expensive
uint256 result = value * 8;
uint256 half = value / 2;

// Cheaper
uint256 result = value << 3;
uint256 half = value >> 1;

Design Patterns for Gas Efficiency

Batch Operations

Combine multiple operations into a single transaction to amortize the base transaction cost (21,000 gas). For example, batch token transfers instead of individual ones.

Lazy Evaluation

Defer expensive computations until they are actually needed. Instead of computing values on write, compute them on read if reads are less frequent.

Events Over Storage

If data only needs to be accessible off-chain, emit events instead of writing to storage. Events cost significantly less gas than storage writes:

// Expensive: stores in contract state
mapping(address => uint256[]) public userActions;

// Cheaper: emit event for off-chain indexing
event ActionPerformed(address indexed user, uint256 actionId, uint256 timestamp);

Tools for Gas Analysis

Several tools can help identify gas optimization opportunities:

  • Hardhat Gas Reporter: Provides gas usage reports for each function call in your tests.
  • Foundry: Forge includes built-in gas snapshots and reports.
  • Remix IDE: Shows gas estimates for transactions in its debugger.
  • Slither: Static analysis tool that can detect certain gas-inefficient patterns.
  • EVM Codes: Reference for opcode gas costs at evm.codes.

Conclusion

Gas optimization is both an art and a science. While micro-optimizations like using prefix increments or unchecked blocks can save small amounts per operation, the biggest gains typically come from architectural decisions—minimizing storage operations, using efficient data structures, and designing contracts that batch operations effectively. Always profile your contracts with gas reporting tools and prioritize optimizations that have the largest impact on real-world usage patterns. Remember that readability and security should never be sacrificed for marginal gas savings.
