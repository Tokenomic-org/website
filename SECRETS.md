## 2. How to set them

### Cloudflare (Workers Secrets â€” encrypted, per-Worker)

```bash
# From repo root, with wrangler logged in (`npx wrangler login`):
cd workers/api-worker
npx wrangler secret put JWT_SECRET            # paste value when prompted
npx wrangler secret put CF_ACCOUNT_ID
npx wrangler secret put CF_API_TOKEN
npx wrangler secret put OAUTH_TOKEN_ENC_KEY   # `openssl rand 32 | base64 | tr -d '=' | tr '+/' '-_'`
npx wrangler secret put INVITE_HMAC_KEY       # `openssl rand -hex 32`
npx wrangler secret put TURNSTILE_SECRET_KEY
# â€¦ repeat for every Cloudflare-runtime secret listed in Â§1a.

# Public [vars] live in wrangler.toml â€” edit and redeploy:
npx wrangler deploy
```

To **list** what's set on a Worker (names only, never values):
```bash
npx wrangler secret list
```

To **rotate**, re-run `wrangler secret put <NAME>` with the new value, then redeploy.

### GitHub Actions (CI secrets)

GitHub UI: **Settings â†’ Secrets and variables â†’ Actions â†’ New repository secret**.

CLI: `gh secret set CLOUDFLARE_API_TOKEN` (paste when prompted).

Reference them in workflows as `${{ secrets.NAME }}`. Today's workflows (`jekyll.yml`, `a11y-perf.yml`) reference none.

### Replit (workspace dev only)

Click the lock icon in the left sidebar â†’ **Secrets** â†’ New secret. Replit injects them as `process.env.NAME` automatically. **Do not** put production runtime secrets here â€” they belong on Cloudflare.

### Local `.env`

Copy the template:
```bash
cp .env.example .env
# Fill in real values. .env is in .gitignore â€” never commit it.
```

## 4. What changed in the codebase from this audit

Nothing functional. The audit confirmed:

- **Workers already read every secret from `c.env.*` / `env.*`.** No `process.env.*` access on the edge.
- **Frontend bundles read zero secrets.** Public config (RPC URLs, WC project id) is injected at build time via `esbuild --define`; nothing else.
- **CI workflows reference no secrets today.** Passing public RPC URLs through `env:` is intentional.
- **`server.js` and Hardhat scripts** correctly use `process.env.*` â€” they only run locally.

The only edits this audit produced are documentation:
- this file (`SECRETS.md`),
- a complete `.env.example` covering every developer-machine variable,
- a "Secrets management" pointer in `replit.md`.
