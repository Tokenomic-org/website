Upgradeable Proxy Contracts

Estimated reading time: 11 minutes

One of the fundamental properties of blockchain is immutability—once a smart contract is deployed, its code cannot be changed. While this provides strong security guarantees, it also presents challenges when bugs are discovered or new features need to be added. Upgradeable proxy contracts solve this problem by separating a contract's storage from its logic, allowing developers to update the logic while preserving the contract's state and address. This article explores the major proxy patterns, their implementation details, security considerations, and best practices.

Why Upgradeability?

Smart contracts handle billions of dollars in value, and real-world deployments often require updates for several reasons:

  • Bug Fixes: Critical vulnerabilities discovered post-deployment need immediate remediation.
  • Feature Additions: Protocols evolve, and new functionality may be needed.
  • Gas Optimization: Improved implementations can reduce transaction costs.
  • Regulatory Compliance: Changing legal requirements may necessitate contract updates.

Without upgradeability patterns, the only option would be to deploy an entirely new contract and migrate all state data—an expensive, error-prone, and disruptive process.

The Proxy Pattern

The proxy pattern uses Solidity's delegatecall opcode to forward function calls from a proxy contract to an implementation (logic) contract. The key insight is that delegatecall executes the implementation's code in the context of the proxy's storage, meaning all state changes happen on the proxy contract.

// Simplified proxy contract
contract Proxy {
    address public implementation;

    fallback() external payable {
        address impl = implementation;
        assembly {
            calldatacopy(0, 0, calldatasize())
            let result := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
            returndatacopy(0, 0, returndatasize())
            switch result
            case 0 { revert(0, returndatasize()) }
            default { return(0, returndatasize()) }
        }
    }
}

This separation means:

  • The proxy holds the contract's address and state (storage).
  • The implementation holds the business logic.
  • Users interact with the proxy address, which never changes.

Transparent Proxy Pattern

The Transparent Proxy Pattern, popularized by OpenZeppelin, addresses a critical issue: function selector clashing. If the proxy and implementation contracts have functions with the same selector, ambiguity arises about which function should execute.

The transparent proxy solves this by enforcing a simple rule:

  • The admin can only call proxy management functions (like upgradeTo).
  • All other callers have their calls delegated to the implementation.
function _fallback() internal {
    if (msg.sender == admin) {
        // Handle admin functions directly
    } else {
        // Delegate to implementation
        _delegate(implementation);
    }
}

Pros: Simple mental model, clear separation of admin and user calls.
Cons: Every call requires an admin check (extra gas cost), admin cannot interact with the implementation as a regular user.

UUPS (Universal Upgradeable Proxy Standard)

The UUPS pattern (EIP-1822) moves the upgrade logic from the proxy to the implementation contract. The proxy is minimal—it only contains the delegatecall forwarding logic and a storage slot for the implementation address.

// UUPS Implementation
contract MyContractV1 is UUPSUpgradeable {
    uint256 public value;

    function initialize(uint256 _value) public initializer {
        value = _value;
    }

    function _authorizeUpgrade(address newImplementation) internal override onlyOwner {}
}

Pros: Cheaper deployment and calls (no admin check in proxy), smaller proxy bytecode.
Cons: If the implementation doesn't include upgrade logic, the contract becomes permanently non-upgradeable. This is a security risk if developers forget to include the upgrade mechanism.

Beacon Proxy Pattern

The Beacon Proxy pattern is designed for scenarios where multiple proxy instances need to share the same implementation. Instead of each proxy storing its own implementation address, all proxies point to a single "beacon" contract that stores the implementation address.

contract Beacon {
    address public implementation;

    function upgrade(address newImplementation) external onlyOwner {
        implementation = newImplementation;
    }
}

contract BeaconProxy {
    address public beacon;

    fallback() external payable {
        address impl = Beacon(beacon).implementation();
        // delegatecall to impl
    }
}

Upgrading the beacon's implementation address simultaneously upgrades all proxy instances—a powerful feature for protocols that deploy many instances of the same contract (like individual user vaults or lending pools).

Diamond Pattern (EIP-2535)

The Diamond pattern extends the proxy concept to support multiple implementation contracts (called "facets"). A mapping of function selectors to facet addresses determines which implementation handles each function call.

This pattern is useful for complex contracts that exceed the 24KB contract size limit, as logic can be split across multiple facets. However, it adds significant complexity and requires careful management of function selectors and storage layouts.

Storage Layout Considerations

One of the most critical aspects of upgradeable contracts is maintaining consistent storage layouts between versions. Since the proxy's storage is used by all implementation versions, changing the order or type of state variables can corrupt data.

Avoiding Storage Collisions

Proxy patterns use specific storage slots (defined by EIP-1967) to store admin and implementation addresses, preventing collisions with implementation storage:

// EIP-1967 slots
bytes32 constant IMPLEMENTATION_SLOT = bytes32(uint256(keccak256('eip1967.proxy.implementation')) - 1);
bytes32 constant ADMIN_SLOT = bytes32(uint256(keccak256('eip1967.proxy.admin')) - 1);

Rules for Upgrade-Safe Contracts

  • Never remove or reorder existing state variables.
  • Only append new state variables at the end.
  • Never change the type of an existing variable.
  • Use storage gaps in base contracts to reserve space for future variables.
  • Use initializer functions instead of constructors (constructors only run in the implementation's context, not the proxy's).
contract MyContractV1 {
    uint256 public value;
    address public owner;
    uint256[48] private __gap; // Reserve 48 slots for future use
}

contract MyContractV2 {
    uint256 public value;
    address public owner;
    uint256 public newVariable; // Uses one gap slot
    uint256[47] private __gap; // 47 remaining slots
}

Security Considerations

Initialization

Since constructors don't work with proxies, initialization functions must be used—and they must be protected against being called multiple times using an initializer modifier.

Access Control for Upgrades

Upgrade functions must be strictly access-controlled. Common patterns include requiring a multi-sig wallet, a timelock, or governance approval before upgrades can be executed.

Implementation Contract Security

The implementation contract should be initialized or destroyed after deployment to prevent attackers from taking ownership of the implementation directly (which could allow them to selfdestruct it in pre-Cancun EVM versions).

Tools and Frameworks

  • OpenZeppelin Upgrades Plugins: Provides Hardhat and Foundry plugins for deploying and managing upgradeable contracts with built-in safety checks.
  • OpenZeppelin Defender: Platform for managing upgrades, access control, and monitoring.
  • Foundry: Supports upgradeable contract testing with its fork and snapshot capabilities.

Conclusion

Upgradeable proxy contracts are an essential tool in the smart contract developer's toolkit, enabling the evolution of on-chain systems while maintaining continuity for users. The choice between Transparent Proxy, UUPS, Beacon, or Diamond patterns depends on the specific requirements of your protocol—whether you prioritize simplicity, gas efficiency, multi-instance upgrades, or modular architecture. Regardless of the pattern chosen, maintaining strict storage layout discipline, implementing robust access controls, and thoroughly testing upgrade paths are non-negotiable practices for secure upgradeable contracts.
